# Portfolio Automation System

## Project overview
Azure-native automated portfolio analysis and paper trade execution pipeline. Single-user personal system. NOT for live trading. All trade decisions require human approval via Teams adaptive card.

> **Open work / where we left off:** see [`FOLLOWUPS.md`](FOLLOWUPS.md) at the repo root.

## Architecture decisions (do not deviate without discussion)
- **Azure AI Foundry** for Claude API ‚Äî project: Portfolio-Analysis, resource: resource-portfolio-analysis (East US 2, Claude not available in East US). API key auth (FoundryApiKey in KV). Endpoints in Function App settings: FOUNDRY_ENDPOINT, FOUNDRY_OPENAI_ENDPOINT. Model: claude-sonnet-4-6, temp 0.2
- **Alpaca paper account** is the canonical source of truth for portfolio positions and balances (`portfolio.positions[]`, `portfolio.balances`, `paper_account` block). Also used for Phase 2 paper-trade execution. REST API, no VM needed. (E*TRADE was removed in commit `bc60604` ‚Äî its OAuth-1.0a tokens expired daily and the integration was dropped; `src/shared/clients/etrade.py` is dead code retained only as historical reference.)
- **FMP (Financial Modeling Prep)** for fundamentals, earnings, ETF look-through, EOD prices, stock news, and fallback congressional trades (senate/house) ‚Äî free tier, 250 req/day
- **FRED** for macro indicators (35 series collected, including 6 labor-market series feeding `labor_signals` scorecard; `CHPMINDXM` deprecated ‚Äî to revisit) ‚Äî free, no practical limit
- **Quiver Quantitative** (primary alternative-data source) for congressional trades, lobbying, government contracts, wikipedia attention ‚Äî hobbyist tier, `Authorization: Token <key>` header
- **Finnhub** for financial market news and company news ‚Äî free tier, 60 calls/min
- **Azure AI Search** (Free tier) for semantic memory recall ‚Äî Phase 1.5, after 60-90 days of data

`swa-storage-connection-string`, `swa-func-master-key`, `github-learning-pat` ‚Äî the `web/api` managed-function app settings (SWA hardening batch, FOLLOWUPS #2, 2026-07-11; the PAT added by the Learning Loop batch, FOLLOWUPS #13/#32, 2026-07-12). These are resolved at **DEPLOY TIME**, not runtime: `az bicep`'s `getSecret()` (`infra/main.bicep`, against `kv-pfauto-prod` with `enabledForTemplateDeployment: true`) bakes the current value into a plain `staticwebapp.bicep` app setting on every infra deploy. (Note: `MassiveApiKey` and the four `Etrade*` secrets may still exist in KV as soft-deletable leftovers from deprecated integrations ‚Äî safe to delete manually. The collector no longer fetches them.)

## Data flow ‚Äî Phase 1
1. Timer fires collector at 09:00 ET weekdays (NCRONTAB: `0 0 9 * * 1-5`, ET-local via `TZ=America/New_York` app setting ‚Äî no DST drift). **NOTE**: this is Linux Consumption, so the IANA `TZ` setting is required. The Windows-only `WEBSITE_TIME_ZONE` is silently ignored on Linux and caused crons to fire 4.5h early (at 09:00 UTC = 05:00 ET) until commit 6f42f1a.
2. Collector reads secrets from Key Vault via Managed Identity
3. Collector calls Alpaca paper account (positions + balances drive the canonical `portfolio.positions[]` / `portfolio.balances`; `paper_account` block also retained with cash, buying_power, equity, last_equity, position list ‚Äî falls back to `src/config/portfolio.json` if Alpaca unreachable; NOTE `paper_account.positions[]` uses the Alpaca-native `qty` field, while `portfolio.positions[]` uses `quantity` ‚Äî code reading positions must accept both, see the 2026-07-07 held_qty incident, FOLLOWUPS #33), FMP (fundamentals, earnings, EOD prices ‚Äî the price-fetch universe is held tickers ‚à™ every role's `selected` incumbent (`shared/quadrants.selected_core_members()`, 2026-07-13 audit finding 1 ‚Äî an unheld reference target previously had no price, no gap row, and no way for band enforcement to synthesize its buy) ‚à™ the ETF watchlist ‚à™ flex candidates, assembled by `collector._build_price_universe()` ‚Äî, ETF data, stock news, fallback congressional), FRED (35 macro series, deep-history for bond + labor scorecards), Quiver (congressional trades, lobbying, gov contracts ‚Äî **lobbying and gov_contracts are filtered client-side to portfolio tickers ‚à™ ETF watchlist and last 90 days; raw responses are ~20K rows/~16 MB and would blow past Claude's 1 M-token limit**), Finnhub (market news, company news)
4. Collector writes full JSON snapshot to `daily-snapshots/YYYY-MM-DD.json` blob
5. Collector writes denormalized rows to 6 Table Storage tables (PortfolioHistory, FundamentalsHistory, MacroHistory, ETFLookthroughHistory, SentimentHistory, TradeHistory)
6. Blob trigger fires analyzer
7. Analyzer reads today's snapshot + queries tables for historical trends + loads last 5 reports for continuity
8. Analyzer calls Claude via Foundry (Sonnet 4.6, temp 0.2, max_tokens 8000). Responses are **streamed** (`shared/clients/foundry.py`) to survive Azure's ~4-min outbound idle drop on long generations (PR #7, 2026-07-02).
9. Analyzer parses response: markdown report + structured trade recommendations JSON. Then, deterministically and post-model: validates `overrides[]` (Tier-2, `shared/overrides.py` ‚Äî session 2026-07-15 Task E1: each record's `direction` is now DERIVED from the sleeve's block + gap sign via `shared/reference_execution.derive_override_direction`, not trusted from the model's claim; a disagreement is corrected-and-flagged, never rejected outright, and both the effective `direction` and the model's original `declared_direction` persist to OverrideHistory). Then **two-pass Tier-1 validation around band enforcement** (session 2026-07-15 Task B1 ‚Äî replaces a single pass that let a soon-to-be-rejected buy's notional starve enforcement cash, see "Deployment lessons" below): **pass 1** runs the **Tier-1 trade validator** (`shared/trade_validation.validate_trades` ‚Äî V1 gate/roster, **V1.5 selected-member-only buys** (a BUY of a role's pool member that isn't that role's `selected` incumbent is rejected ‚Äî the role-based roster otherwise lets a non-selected pool member, e.g. SOXX when `semis.selected = SMH`, pass straight through; the lone exception is `intl_leader`, whose `selected` follows `intl_governance.leader_pick` automatically), V2 exemption, V3 window (now also rejecting a BUY with no matching reference/gap row outright, rather than silently skipping the window), V4 mechanical) over the model's raw trades and drops what it would reject; **band enforcement** (`shared/reference_execution.reconcile` ‚Äî synthesizes de-risk shortfall trades toward reference, now also treating a `LEGACY_EXITS` sell as de-risk, Task D1) runs against the pass-1 SURVIVORS (so a rejected buy's dollars are never mistaken for spent cash) and now counts an off-roster held name's sell proceeds toward available cash too (Task B2) while still never targeting an off-roster name for synthesis; **pass 2** re-validates the full merged list (survivors + synthesized trades) so cumulative checks see the final list. Rejections from both passes combine into the report addendum, deduped by trade id. Either pass crashing sets `validation_error` and the auto-executor refuses the file (fail-closed).
10. Writes report to `daily-reports/YYYY-MM-DD.md`, trades to `daily-trades/YYYY-MM-DD.json` (with `trade_validation`/`reference_execution` blocks and an optional `watch_candidates` key), OverrideHistory rows. **`watch_candidates` contract:** optional top-level key in `daily-trades/{date}.json` ‚Äî `[{"symbol": "XYZ", "reason": "one line"}, ...]`, max 6 entries, sanitized by `_split_response` (malformed entries dropped with a log line, never fatal; the trades file itself is never invalidated by a bad entry). The executor and trade validator ignore this key entirely ‚Äî it is consumed only by the NEXT day's collector run to seed dynamic flex candidates. Each `flex_candidates` FMP profile in the snapshot's `flex_candidates[]` block carries `source: "static"|"dynamic"` indicating whether the name came from `flex-candidates.json` or from the analyzer's prior `watch_candidates` emission.
11. Outputs surfaced in `swa-pfauto` (no Logic App delivery; user pulls report via web UI). Optional email/OneDrive copies can be added later if needed.

## Data flow ‚Äî Phase 2 (paper execution)
There are TWO execution paths into the same `executor.execute_approvals`:

**Auto path (currently live ‚Äî `AUTO_EXECUTE_ENABLED=true`, paper only):**
1. `auto_executor` timer fires 09:35 ET; retry nets at 10:05 + 11:05 ET (`auto_executor_retry`, #29) cover variable analyzer latency and missed opens
2. Reads `daily-trades/{ET-today}.json` (ET trading date via `shared/timeutil.today_et` ‚Äî never UTC), treats every recommendation as approved, but **refuses** any file flagged by the Tier-1 validator (`validation_error`, rejected-stamped trades). Gated by the Alpaca market clock (defers if closed). Idempotent by a cache asymmetry (terminal outcomes cached; failures left uncached so retries re-attempt) + date-scoped `client_order_id`.
3. Executor connects to Alpaca paper API, places orders (sells before buys), applies the defensive sell filter (drops sells of unheld symbols; trims oversize qty). Writes `daily-executions/{date}.json` + TradeHistory.

**Manual path (Static Web App ‚Äî for when auto is off or for live trading later):**
1. User signs into `swa-pfauto` with Entra ID work account (Easy Auth; allowed-role `owner`)
2. `/today` renders the report + trade table with per-row checkboxes
3. User selects 1‚Ä¶N trades and clicks **Approve selected**
4. SWA managed API (`web/api/`) records the decision in `approvals/YYYY-MM-DD.json` and calls the `func-pfauto` executor endpoint (function master key fetched via SWA MI ‚Üí Key Vault reference)
5-6. Same executor ‚Üí same `daily-executions/{date}.json` + TradeHistory; SWA `/today` polls and shows status inline

## Learning Loop (monthly strategy review + human-approval amendment channel)
**Spec: `docs/specs/Learning_Loop_v1.0.md`. Resolves FOLLOWUPS #13 + #32.** Where the
daily analyzer executes toward a fixed reference and grades its own trades, the
Learning Loop is the missing half: a monthly, high-reasoning review that proposes
*changes to the rules themselves* from the graded record, gated entirely behind human
approval. **Non-negotiable invariant: proposer ‚âÝ approver.** Nothing in this system can
apply a change directly ‚Äî every approved proposal becomes a GitHub pull request that a
human merges after CI passes; the automation credential cannot merge (branch protection,
not convention).

- **`src/learning/bundle.py`** assembles the reviewer's input: the trailing 35 days of
  daily reports (oldest dropped first if `LEARNING_BUNDLE_MAX_TOKENS` is exceeded ‚Äî graded
  records and live config NEVER drop), capture-fine TradeHistory, every OverrideHistory
  layer, and live config fetched from **GitHub raw at master HEAD** (not the deployed
  function package ‚Äî a proposal diff must apply to master) alongside the recorded
  `diff_base_sha`. `fetch_override_history` (session 2026-07-17, Task G-b) annotates each
  `layer: "override"` accepted/downsized row with `_direction_suspect` ‚Äî true when
  `declared_direction` is absent, which is possible ONLY for a record from before the
  2026-07-15 Task E1 derived-direction fix (every accepted/downsized decision since then
  always populates both `direction` and `declared_direction`). A pre-E1 row's `direction`
  may be backwards (07-15 filed three de-risk holds ‚Äî GLD/XLP/TLT ‚Äî as `re_risk`), which
  would invert the sign Phase-5's counterfactual grading uses; the monthly reviewer must
  not treat a flagged row's direction/outcome as reliable signal without accounting for it.
- **`src/learning/handler.py`** (`learning_reviewer` timer, Saturdays 12:00 ET with an
  in-code first-Saturday-of-month gate; `learning_run` HTTP trigger for the manual
  button, 1/day rate-limited) calls the reviewer model ‚Äî **`LEARNING_MODEL`, default
  `claude-sonnet-4-6`, the analyzer's EXISTING Foundry deployment** (no new deployment,
  no new quota; upgrading to Claude Fable 5 is a config flip once quota lands, FOLLOWUPS
  #40) ‚Äî then validates the output before anything is stored as pending.
- **`src/learning/schema.py`** + **`src/learning/diffcheck.py`** (pure-Python unified-diff
  apply checker, no third-party dep) deterministically enforce the proposal contract:
  4 classes (0 Observation / 1 Prompt-text / 2 Parameter, needs `evidence_n >= 10` graded
  rows / 3 Structural, no diff ‚Äî a spec draft + implementation brief instead), a
  **target-file allowlist** (`project-instructions.md`, `risk-limits.json`,
  `sleeve-roles.json`, `flex-candidates.json` ‚Äî code/validator/infra are never a diff
  target), hard caps (‚â§3/cycle, ‚â§1 structural, reverts exempt), and the forced re-review
  rule (every amendment past its `review_by` date must get an explicit keep/revert/amend
  this cycle). **Any violation fails the whole cycle loudly** (`failed_validation`, raw
  output preserved) ‚Äî nothing partially surfaces.
- **`web/api/learning_github.py`** is the ONLY thing that writes to GitHub, and only via
  PR: Approve re-validates a class 1-2 diff against CURRENT master (stale ‚Üí the proposal
  regenerates next cycle, never force-applied), then branches, commits, and opens a PR
  with the proposal JSON verbatim in the body. Merge detection is lazy (checked on every
  `GET /api/learning/proposals`, cached ‚â•10 min) ‚Äî no webhook in v1. On merge, the
  proposal flips to `applied` and an OverrideHistory row (layer `amendment`) is written
  so the loop eventually grades itself (mechanical grader deferred to FOLLOWUPS #38,
  ‚â•5 amendments).
- **Learning tab** (`web/learning.html`/`.js`) ‚Äî nav entry rendered ONLY when
  `GET /api/learning/proposals` reports `LEARNING_PHASE >= 2` (`app.js::renderLearningNav`).
  Decision buttons render only at phase 3. **3-phase rollout** (`LEARNING_PHASE` app
  setting, ships `1`): 1 = dry-run (reviewer runs, no tab) ‚Üí 2 = tab read-only ‚Üí 3 = full
  loop. `POST /api/learning/decision` and `/run` require the authenticated principal to
  hold the `owner` role (the same role the platform's route rules already enforce), with
  an OPTIONAL tighter pin to one specific SWA user id via `OWNER_USER_ID` ‚Äî that's SWA's
  own opaque `userId` from `/.auth/me` *after* signing in, **not an Entra object id**
  (the two are unrelated identifiers; a wrong value here silently denies everyone,
  including the real owner ‚Äî fixed 2026-07-12 after exactly that shipped as
  `OWNER_OBJECT_ID` in PR #23's first draft). Empty/unset `OWNER_USER_ID` = roles-only
  mode.
- **Tables**: `LearningCycles` (one row per cycle, status/model/mode/narrative) and
  `LearningProposals` (one row per proposal, `status`: pending ‚Üí approved/rejected/stale
  ‚Üí applied) ‚Äî see Table Storage schemas below.

## Table Storage schemas (10 tables)
- **PortfolioHistory**: PK=ticker, RK=date ‚Äî positions, weights, P/L
- **FundamentalsHistory**: PK=ticker, RK=date ‚Äî P/E, DCF, rating, earnings date
- **MacroHistory**: PK=series_id, RK=date ‚Äî FRED values with deltas
- **ETFLookthroughHistory**: PK=etf_ticker, RK=date ‚Äî holdings, country/sector allocation
- **SentimentHistory**: PK=date, RK=indicator ‚Äî VIX, spreads, P/C ratios, percentiles
- **TradeHistory**: PK=year-month, RK=trade_id ‚Äî full lifecycle from recommendation to 30/60/90d outcome. Phase C adds (write-once at recommendation, flex trades) the ¬ß7 reasoning enums `primary_trigger`/`thesis_type`/`trigger_evidence`/`catalyst_date`, and (stamped later by the collector at maturity) `price_at_rec`/`spy_at_rec`/`ret_Nd_pct`/`spy_ret_Nd_pct`/`excess_Nd_pp`/`call_correct_Nd`/`outcome_status`. Keys are **lowercase** across analyzer + executor + collector writes ‚Äî Azure Tables is case-sensitive and upserts MERGE onto one entity (Phase C ¬ß9 casing fix).
- **OverrideHistory**: PK=year-month, RK=`OV-YYYYMMDD-NNN` ‚Äî one row per override record the analyzer emitted (Phase 4, write-once): `outcome` (accepted/downsized/rejected), `validator_reasons`, `premise_challenged`, `sleeve` (V1_1), `direction`, `magnitude_pp`, `downsized`, `evidence`, `evidence_count`, `falsifier`, `falsifier_date`, `clean_data_only`, `enforced` (Finding-2 band enforcement fired on this sleeve). **Phase 5 (merged 2026-07-05) now STAMPS** `outcome_status`/`resolved_correct` at each record's `falsifier_date`, graded against the reference-path counterfactual ("did disagreeing beat obeying"); the collector maturity pass writes them. Also holds the roster-revision **`sleeve_switch`** / **`intl_leader_rotation`** rows (Task G, RK `SW-‚Ä¶`/`ILR-‚Ä¶`) graded vs the incumbent counterfactual at 30/60/90d, and the **`regime_suspect`** rows (FOLLOWUPS #12 Task C, RK `RS-YYYYMMDD-{bucket}`, one per suspect bucket per report day: `sleeve` = the bucket e.g. `Q3`, `favored_streak`, `streak_excess_pp`, `action` = increased/held/reduced that session ‚Äî **not yet graded by any stamper**, see `quadrant_performance` above). The Learning Loop (FOLLOWUPS #13/#32) adds the **`amendment`** layer (RK `AMD-{proposal_id}`, written by `web/api` when a proposal's PR merges): `proposal_id`, `sleeve` = the amendment's `target_file`, `falsifier`, `review_by` ‚Äî grading hooks also null until the mechanical amendment grader (FOLLOWUPS #38) is built.
- **SleeveSelectionState**: PK=`state`, RK=role_id (or `intl_governance`) ‚Äî hysteresis streak (`challenger`/`streak`/`selected`) for the `sleeve_selection` scorecard + the intl `leader`/`composite` de-rotation state.
- **LearningCycles**: PK=year-month, RK=cycle date (`YYYY-MM-DD`) ‚Äî one row per Learning Loop reviewer run: `trigger` (timer/manual), `status` (completed/failed_validation), `model`, `mode` (full/observation_only), `proposal_count`, `narrative`.
- **LearningProposals**: PK=year-month (of its cycle), RK=proposal id (`AMD-YYYY-MM-DD[-N]`) ‚Äî one row per proposal: `class`, `title`, `change_summary`, `data_summary`, `target_file`, `diff`, `evidence`, `evidence_n`, `expected_effect`, `falsifier`, `review_by`, `spec_draft`/`implementation_brief` (class 3), `re_review_of`/`is_revert`, `diff_base_sha`, and the decision lifecycle: `status` (pending ‚Üí approved/rejected/stale ‚Üí applied), `decision`/`decision_reason`/`decided_at`/`decided_by`, `pr_url`/`pr_number`/`pr_last_checked`, `applied_at`.

**Override protocol + enforcement** (responsiveness brief Phase 4/Finding 2/#28 ‚Äî the phases that **change report behavior and make the reference enforceable**). The analyzer prompt **consumes `reference_weights` / `divergences` / `transition_watch`** and executes toward the reference: Section 2 has a **Reference column** and **Current-vs-Reference gap**, and **Recommended = Reference ¬± logged overrides**. Deviating ‚Äî *including a "hold" of a sleeve more than `gap_band_pp` off reference* ‚Äî requires a structured **`overrides[]`** record (**OVERRIDE_SCHEMA_V1_1**, with a mandatory per-sleeve **`sleeve`** field): `{sleeve, premise_challenged, direction (de_risk|re_risk), magnitude_pp, evidence[], falsifier, falsifier_date, clean_data_only}`. `analyzer.assert_override_prompt_schema` gates the prompt on load (sentinels `OVERRIDE_SCHEMA_V1_1`/`reference_weights`/`overrides`), mirroring the flex gate. Three deterministic layers run post-model:
- **Tier-2 override validation** (`shared/overrides.py::validate_overrides`): structural gates (sleeve + falsifier + date + clean non-empty evidence + within `max_magnitude_pp` band + valid direction) reject; the **de-risk/re-risk asymmetry** (spec ¬ß6) accepts a de-risk override on one clean item, **downsizes** an under-evidenced re-risk override, **rejects** one with no evidence. **Direction is derived, not trusted** (session 2026-07-15, Task E1): `derive_override_direction` (`shared/reference_execution.py`, shares the block model with `is_de_risk_move`) computes the direction from the sleeve's block + the gap sign (current % ‚àí reference %) ‚Äî a Damper/SGOV sleeve held ABOVE reference (or an Amplifier/`LEGACY_EXITS` sleeve held BELOW) is `de_risk`; the opposite is `re_risk`. The DERIVED direction ‚Äî never the model's raw claim ‚Äî governs the asymmetry bar; a disagreement is corrected-and-flagged (a reason is appended, the override is not rejected for it). Both the effective `direction` and the model's original `declared_direction` persist to OverrideHistory (2026-07-14 correctly filed a GLD-above-reference hold as `de_risk`; 2026-07-15 filed the identical case ‚Äî plus XLP/TLT ‚Äî as `re_risk`, backwards, which would have imposed the harder evidence bar on a cheap override, and would have inverted the sign `_grade_override`'s Phase-5 counterfactual grading uses).
- **Band enforcement** (Finding 2, `shared/reference_execution.py::reconcile`): D1 an override caps the *residual* not the move (`required_move_total = max(0, |gap| ‚àí max(allowed_residual, gap_band_pp))`); D2 tranche pace (`required_move_today = min(required_move_total, tranche_pp_max)`); D3 a de-risk shortfall (sell overweight amplifier **or a `LEGACY_EXITS` name** / buy underweight damper-or-SGOV ‚Äî the `LEGACY_EXITS` sell leg added session 2026-07-15 Task D1, since slow-walking a wind-down exit is unambiguously de-risk) is **synthesized** as a `source:"band_enforcement"` trade ‚Äî re-risk shortfalls are only flagged `non_compliant_flagged` (never auto-added), and (session 2026-07-15 Task D2) surfaced in a dedicated report addendum (`analyzer._flagged_sleeves_addendum`), not just the JSON. This kills the 2026-06-30 silent-hold pathology (the model could previously just not-trade an out-of-band sleeve). `reconcile`'s cash accounting counts an off-roster held name's sell proceeds toward `cash_avail` (Task B2) while still never targeting an off-roster name for synthesis, and ‚Äî since session 2026-07-15 Task B1 ‚Äî runs against the Tier-1 validator's pass-1 SURVIVORS rather than the model's raw trades, so a buy the validator will reject can never be mistaken for already-spent cash (see "Data flow ‚Äî Phase 1" step 9 above). **Gap-row pricing is paper-canonical for held names** (session 2026-07-17, Task A): `_build_reference_gaps`'s `_price` helper prefers the paper-account position's `current_price` for any HELD symbol ‚Äî the same price `current_pct` is itself computed from ‚Äî falling back to the FMP EOD close only for an unheld reference target (no position price exists). Before this fix FMP could win outright for a held name, and any FMP-vs-paper price divergence over ~3% mixed a paper-priced `current_pct` with an FMP-priced V3 landing calculation and could phantom-clamp a legitimate full exit to a 1-share stub (2026-07-16 MU incident: a 5.9% divergence clamped 2‚Üí1 share; a 2.8% divergence the next day happened to slip under the validator's epsilon and pass ‚Äî luck, not correctness). A permanent coherence invariant is pinned in `tests/test_price_basis_coherence.py`: for every gap row with `held_qty > 0`, `current_pct` must equal `held_qty * price / equity * 100` within rounding.
- **Tier-1 trade validator** (#28, `shared/trade_validation.py::validate_trades`, runs AFTER the reconcile merge): V1 gate/roster (no amplifier buy while gate closed ‚Äî **except an equal-weight within-role substitution funded by selling the same role's old member, which is regime-neutral and passes**; core-only in `trades[]`; **any buy of a LEGACY_EXITS name is rejected** ‚Äî core re-entry closed, flex only), **V1.5 selected-member-only buys** (roster_revision_2026-07 seam fix): every pool member of a role reads as an ordinary `CORE_ROSTER` name to V1/V3, so a BUY of a non-selected pool member (e.g. SOXX when `semis.selected = SMH`) would otherwise pass straight through ‚Äî a near-zero reference row lands inside the ¬± band window, and with no gap row V3 used to skip the window entirely. V1.5 rejects any BUY where `role_of(symbol)` is not None and the symbol isn't that role's `selected` incumbent ("non-selected pool member ‚Äî selection changes go through sleeve-roles.json"), **except** the `intl_leader` role, whose `selected` follows `intl_governance.leader_pick` automatically (Task F) ‚Äî a buy of the *current* `leader_pick` (passed in via `quadrant_ctx["intl_leader_pick"]`, sourced from the snapshot's `intl_governance.leader_pick`) passes even before `sleeve-roles.json` is committed to match. SELLs of non-selected pool members are unaffected (substitutions, unwinds). V2 exemption (retired to a no-op ‚Äî `EXEMPT_HOLDS=()`; machinery kept), V3 window (post-trade weight inside `reference ¬± max(residual, band)`, floor-protected ‚Äî **legacy exits AND non-selected pool members bypass the floor so an exit sell reaches 0** (2026-07-13 audit finding 2, B0 decided: sell-to-zero ‚Äî `_non_selected_pool_member()` mirrors V1.5's role/leader_pick logic exactly, so a held non-selected pool member like EWZ/VSS/IEMG/IDMO/EWJ is no longer a permanent 0.1%/1-share dust stub; the bypass is never keyed on `reference_pct == 0`, since a *selected* out-of-favor name can legitimately show ref 0 and still owes its floor; the sell-clamp math also floors at 0 rather than occasionally computing a negative share count); **a BUY with no matching gap row is now rejected outright** ("no reference row ‚Äî cannot window the buy") rather than silently skipping the window ‚Äî closes the second half of the same seam; a SELL of a held name without a row keeps its prior behavior; an **off-roster held name** (a flex leftover like MU, 2026-07-13 audit finding 3) gets a `reference_pct: 0.0, off_roster: True` gap row from `_build_reference_gaps` so its SELL is still V3/V4-clamped ‚Äî off_roster rows are filtered out of `reconcile`'s working set so band enforcement never synthesizes a trade against one, and `_post_validation_cash` also falls back to the paper-account position price when no gap row exists) ‚Äî **with a SGOV literal-cash carve-out**: a SGOV BUY funded purely from *pre-trade* literal cash (not same-day sell proceeds) above the `literal_cash_target_pct` buffer is a pure cash-sleeve composition swap (cash sleeve = SGOV + literal cash), so it is exempt from the per-name window and clamped to the buffer budget rather than rejected (fixes the 2026-07-09 idle-cash pathology where SGOV at 28.44% vs a 28.50% window ceiling rejected a $4k cash‚ÜíSGOV conversion); SGOV sells are windowed normally ‚Äî V4 mechanical (sell ‚â§ held, buy ‚â§ cash-after-sells, integer shares, min-notional). Rejected trades move to `rejected[]` (never submitted); clamped trades trimmed to the window edge. **This is what makes "enforced downstream" true** ‚Äî before #28 a hallucinated trade flowed from LLM JSON straight to Alpaca.

An override can never loosen the deployment gate or breach a Tier-1 limit (floor/ceiling/exempt). Override decisions persist write-once to **`OverrideHistory`** (`analyzer._write_override_history`, layer-tagged, PK=year-month); **Phase 5 (merged) stamps their matured outcomes** vs the reference counterfactual. Config: `src/config/risk-limits.json` ‚Üí `override_protocol` (`max_magnitude_pp` 15, `re_risk_min_evidence` 2, `gap_band_pp` 5) + `reference_execution` (`tranche_pp_max` 10, `enforce` true, `enforcement_turnover_max_pct` 20, `min_notional_usd` 115). **Auto-execute is paper-only (`AUTO_EXECUTE_ENABLED=true`); a `validation_error` file is refused by the auto-executor (fail-closed); live trading still requires human approval.**

**`transition_watch`** (responsiveness brief Phase 3) is a deterministic **pre-staging** signal: realized inflation is laggy, so when the Phase-2 `leading_vs_lagging_inflation` divergence is `active`, `collector._build_transition_watch` projects the quadrant the *leading* signal points to and emits a **bounded, partial lean** that `reference_weights` applies ‚Äî **without moving the binding `active_quadrant` / `regime_gate` / realized inflation axis** (spec ¬ß6). It **reuses** the Phase-2 divergence (never re-derives breakevens/oil). Output: `{active, projected_quadrant, direction (de_risk|re_risk), staged_fraction, basis[], status}`. **Asymmetry (the safety):** a *de-risk* transition (projecting a more defensive quadrant) stages readily at the full fraction; a *re-risk* transition (more offensive) needs ‚â•`re_risk_min_confirmations` leading signals agreeing (breakevens AND oil) and a smaller fraction, else it does not activate. Leading data stale/absent ‚Üí `status:"indeterminate"`, no lean. `reference_weights` applies it as a **convex blend** (`(1‚àíf)¬∑base + f¬∑projected`, `f`‚â§0.30 ‚Äî never a full flip) and surfaces it in its `transition_lean` field. Config in `risk-limits.json` ‚Üí `transition_watch`. **Phase 3 changes `reference_weights`** but is still report-inert until Phase 4.

- Human approval required for ALL **live** trade execution ‚Äî no autonomous live trading. **Paper-only auto-execute** is enabled via app setting `AUTO_EXECUTE_ENABLED=true`: a 09:35 ET timer (`auto_executor`, NCRONTAB `0 35 9 * * 1-5`, ET-local via `TZ=America/New_York`) reads `daily-trades/{ET-today}.json` and submits every recommendation to Alpaca paper, gated by Alpaca market clock (defers if closed). **Retry timers** at 10:05 + 11:05 ET (`auto_executor_retry`, #29) cover variable analyzer latency / missed opens; idempotent via a cache asymmetry (terminal outcomes cached, failures not) + date-scoped `client_order_id`; a `no_trades` outcome at ‚â•11:00 ET logs ERROR (analyzer never produced the file). The executor **refuses** any file the Tier-1 validator flagged (`validation_error` or a rejected-stamped trade ‚Äî fail-closed). It also applies a **defensive sell filter**: any sell against a symbol not held in the Alpaca paper account is dropped (status `not_held_in_paper_account`); oversize qty is trimmed. Drops recorded in `daily-executions/{date}.json` `skipped[]`.
- **One-time paper seeder** (`POST /api/seeder`): idempotent mirror of `src/config/portfolio.json` into Alpaca paper. Use `whole_shares_only=true` for off-hours seeding (Alpaca rejects fractional day-orders queued outside market hours ‚Äî we floor qty to int, drop sub-1 tickers). Symbols already held OR with a pending open buy order are skipped (`skipped_already_held` / `skipped_open_order_pending`) so re-runs never duplicate. Per-run report written to `seeding/{utc_ts}.json`.
- Phase 1 must run clean 30+ days before Phase 2 is enabled
- Temperature 0.2 for Claude analysis calls (consistency)
- Sells execute before buys in multi-trade recommendations (free up cash)

## Deployment lessons (hard-won ‚Äî see infra/modules/storage-roles.bicep + .github/workflows/deploy-code.yml)
- Function App MI requires **Storage Account Contributor** on the storage account in addition to Blob Data Owner / Queue Data Contributor / Table Data Contributor. Host startup calls `BlobServiceClient.GetPropertiesAsync()` which needs `blobServices/read`, not in the data-plane roles. Without it: persistent `AuthorizationPermissionMismatch 403`, host faults, zero functions registered.
- Workflow pip install MUST pin manylinux2014 wheels (`--platform manylinux2014_x86_64 --python-version 3.11 --implementation cp --only-binary=:all:`). GitHub `ubuntu-latest` ships GLIBC 2.39; Functions Linux Consumption image is older. Native wheels (e.g. `cryptography`) otherwise fail with `GLIBC_2.33 not found` and the Python worker silently fails to load.
- Deploy model: run-from-package via blob (`WEBSITE_RUN_FROM_PACKAGE=<blob URL>`). `func azure functionapp publish` does not work with identity-based `AzureWebJobsStorage`. App runs in read-only mode ‚Äî portal Test/Run hits CORS; invoke via admin REST instead: `POST https://func-pfauto.azurewebsites.net/admin/functions/<name>` with master key from `az functionapp keys list`.
- Workflow path filter is `src/**`; workflow-only changes need manual `gh workflow run "Deploy function code" --ref master`.
- Quiver `/beta/live/lobbying` and `/beta/live/gov_contracts` return ~20K rows of all-market activity per call (~12 MB + ~4 MB JSON). Collector MUST filter client-side to portfolio tickers ‚à™ ETF watchlist AND last 90 days ‚Äî otherwise the daily snapshot balloons to ~20 MB and the analyzer prompt blows past Claude's 1 M-token context window (observed: 6.2 M tokens ‚Üí permanent 400). See `_row_ticker`/`_row_date` filter in `src/collector/handler.py` after the Quiver fetch. Commit `20cb2b0`.
- `host.json` `functionTimeout` is `00:40:00`. The old `00:10:00` was the **Consumption** plan max; on **Flex Consumption** the cap is far higher and the Foundry client allows up to 600 s/attempt √ó 3 retries, so 10 min could hard-kill the analyzer mid-Claude-call (no `Executed`, no exception, no report ‚Äî just a silent recycle at the 10-min boundary). Large June snapshots (~1.2 MB / ~240K tokens) exposed this. Can also be overridden live without redeploy via the `AzureFunctionsJobHost__functionTimeout` app setting. Commits `743b5ad` (original), 2026-06-09 fix.
- **EventGrid blob-trigger `functionName` (Flex Consumption)**: the webhook URL in `deploy-code.yml` MUST use `functionName=Host.Functions.analyzer`, NOT bare `functionName=analyzer`. The bare form is accepted by `/runtime/webhooks/blobs` (HTTP 200, EventGrid reports DeliverySuccess) but cannot be routed to the registered listener, so the analyzer **never fires** and nothing errors ‚Äî looks healthy end-to-end while silently dropping every event. Also requires the `Microsoft.EventGrid` resource provider to be **`Registered`** (not `Registering`) or storage publishes zero events (PublishSuccessCount=0). Diagnosed 2026-06-09.
- Runtime knobs `TZ=America/New_York`, `AUTO_EXECUTE_ENABLED=true`, and `AzureFunctionsJobHost__functionTimeout=00:40:00` now live **in `functionapp.bicep`** (IaC), so an `infra/**` deploy sets rather than wipes them. They were previously applied post-deploy, and an `az deployment group create` replaces the function app's app-setting set wholesale ‚Äî which silently dropped them after the Flex migration and disabled the pipeline. Without `TZ`, crons run in UTC (collector 09:00 UTC = 05:00 ET; auto_executor 09:35 UTC = pre-market ‚Üí defers forever). `AUTO_EXECUTE_ENABLED=true` in IaC intentionally overrides the "Phase 1 clean 30+ days" gate. Note: setting any app setting restarts the host ‚Äî which is also what (re)connects Application Insights if telemetry has gone silent.
- **Editing `deploy-infra.yml` or `deploy-web.yml` self-triggers that deploy.** Both include their **own filename** in the push path filter (e.g. `paths: ['infra/**', '.github/workflows/deploy-infra.yml']`), so a workflow-only edit to either fires a full deploy on push. `deploy-code.yml` is the exception ‚Äî it filters on `src/**` only, so editing it does NOT trigger (needs manual `gh workflow run`). Consequence observed 2026-06-15: bumping action versions in all three YAMLs triggered an unintended infra + web deploy ‚Äî the infra deploy re-applied unchanged Bicep and, per the SWA secret-wipe issue, wiped the SWA's `STORAGE_CONNECTION_STRING` + `FUNC_MASTER_KEY`. **This whole failure mode is OBSOLETE as of 2026-07-11 (FOLLOWUPS #2, `fix/swa-hardening`)** ‚Äî those two settings are now resolved from Key Vault at deploy time (`main.bicep`'s `getSecret()` calls), so an infra deploy sets the correct current value instead of wiping it; there is no more manual post-deploy runbook step. Also note: portfolio resources live in the **EasyGridsProduction** subscription under the **jgarrote@easygrids.com** identity ‚Äî a DIFFERENT Entra tenant from a Quirch-default az session. With the wrong identity even `--auth-mode login` blob reads fail with `InvalidAuthenticationInfo: Issuer validation failed` (observed 2026-07-07). The working sequence: `az login --use-device-code` as jgarrote@easygrids.com, then `az account set --subscription EasyGridsProduction`. Only then do blob reads, `az functionapp keys list`, and KV secret reads succeed.
- **A stale resting Alpaca order permanently blocks all future orders on that symbol, invisibly.** Diagnosed 2026-07-15 (FOLLOWUPS #42, Task A0): a GTC stop order the flex engine placed on MU 2026-07-08 (as part of its own stop management, never sent through `_place_one` ‚Äî core trades never carry resting stops) was left open (`status: "new"`) after the engine's internal `held[]` state dropped the position, locking both shares as order collateral (`qty_available: 0`). Every subsequent daily-executor sell attempt for MU 403'd ‚Äî silently, since `_place_one` only logged the error and moved on, and nothing read `daily-executions/{date}.json` back the next day to notice. Confirmed via `GET /v2/positions/{symbol}` (`qty_available`) and `GET /v2/orders?symbols={symbol}&status=all`. **Fixed (collision point, 2026-07-15):** the executor now calls `_cancel_conflicting_orders` before every submission (cancels any pre-existing open order on that symbol first ‚Äî today's recommendation supersedes a stale resting one); `execution_review` (Task A1) makes a recurrence visible in the next day's snapshot instead of silent. **Fixed (root cause, session 2026-07-17, Task F):** the flex engine itself now prevents the orphan from ever accumulating ‚Äî `flex/reconcile.py::reconcile_ledger` returns a new `orphan_orders` list (every broker OPEN order for a symbol the ledger doesn't track at all, surfaced in `flex_state.reconcile.orphan_orders`), and `flex/handler.py::_sweep_orphan_orders` cancels its own orphans every ~15-min tick ‚Äî STRICTLY scoped to its own `client_order_id` family (current `FLEXC-` + the legacy pre-split `flex-` naming), never a DayTrade Lab (`FLEXD-`) or daily-executor order. Additionally (**decision G1 = yes**), repair/trail stop orders switched `time_in_force` from `"gtc"` to `"day"` ‚Äî a stop only triggers during regular hours anyway, so DAY gives identical protection with no immortal stale order surviving a future ledger-row loss; it is simply re-placed every in-hours tick by the existing no-naked-long check in `reconcile_ledger`.
